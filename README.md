# House Prices ML

ეს არის kaggle-ს [კონკურსის](https://www.kaggle.com/competitions/house-prices-advanced-regression-techniques?authuser=0) კოდი სადაც მოდელის დავალებაა სახლის ფასი შეაფასოს მოცემული ინფორმაციით.

> mlflow-ს ლოგები გადმოვიტანე სხვა რეპოზიტორიიდან(typo-ს გამო სხვაგან ვლოგავდი) და ჩანს რომ 12 რიცხვში დავლოგე. ზუსტი ლოგების დროების და შესაბამისი მოდელების ნახვა შეიძლება ამ [რეპოზიტორიაში.](https://dagshub.com/adola23/HousePricesML.mlflow/#/)

### მიდგობა
ამ პრობლემის გადასაჭრელად მე ვეცდები გამოვიყენო Linear Regression და Decision Tree რომლებსაც სხვდადასხვანაირად დამუშავებულ მონაცემებს გადავცემ და საუკეთესოს ამოვირჩევ.

#### სტრუქტურა
- model_experiment.ipynb - მთავარი ნაწილი ხდება ამ ფაილში სადაც  Cleaning, Feature Engineering, Feature Selection და Training ის სხვადასხვა მიდგომებს გავტესტავ.
- model_inference.ipynb - ამ ფაილში იქნება საბოლოო test set ზე პროგნოზი, საუკეთესო მოდელის გამოყენებით და ავტვირთავ kaggle-ზე.
- README.md - ქართულ ენაზე აღწერილი პროექტი და ექსპერიმენტების შედეგები



## ექსპერიმენტები
1. ### **BasicLinearRegression1**
    > [ექსპერიმენტის ლინკი](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/1)
    #### **მიმოხილვა** 
    ეს იყო პირველი და სატესტო მოდელი რომლითაც მინდოდა გამეტესტა რომ გარემო გამართული იყო და დავწერე ყველაზე მარტივი ლოგიკა რაც მომაფიქრდა.
    #### **Feature Engineering**
    რიცხვებში NaN-ები ჩავანაცვლე საშუალოებით.
    კატეგორიულები კი უბრალოდ რიცხვად გადავაქციე.
    #### **Feature Selection**
    ყველაფერი დავტოვე ამ ექსმერიმენტისთვის
    #### **Training**
    Linear Regression დავათრეინე საბოლოო y-ებზე
    #### **შედეგი და დასკვნა**
    თავიდან რამდენიმე მეტრიკა დავლოგე მათ შორის
    log-ების rmse-ც(~0.72) და ყველა მეტრიკა საკმაოდ ცუდი იყო. ამის მიზეზის დიდი ნაწილი იქნება რომ ზედმეტი მონაცემი დავტოვე რამაც შემოიტანა noise და ამასთან ერთად შემდეგ მოდელებში გამოჩნდება რომ ლოგარითმების rmse-ს შემცირებაში საკმაოდ დიდი შედეგი მოაქბს y-ის ნაცვლად log(y)-ის შესწავლას.

2. ### **LinearRegression ONE_HOT +  CORRELATION + train on log(y)**
    >[ექსპერიმენტის ლინკი 1](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/3)
    
    >[ექსპერიმენტის ლინკი 2](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/2)
    #### **მიმოხილვა** 
    ამ მოდელში თავიდან ვიყენებდი iv + woe –საც და ცუდი შედეგი მომცა ასერომ კოდი წავშალე და მის გარეშე გავაკეთე. ამ მოდელმა შედარებით უკეთესი შედეგი მოგვცა. ასევე მხოლოდ ლoგარითმების rmse-ს დალოგვა დავიწყე ამის მერე რადგან მხოლოდ მისი მინიმიზაცია არის ჩემი მიზანი.
    #### **Feature Engineering**
    რიცხვებში ისევ საშუალოებით ჩავანაცვლე.
    #### **Feature Selection**
    გადავყარე კატეგორიული ცვლადები რომლსებსაც treshold-ზე მეტი კატეროგია ქონდათ. და კორელაციის ფილტრით გადავყარე treshold–ზე მეტი კორელაციის მქონე სვეტები.
    #### **Training**
    Linear Regression დავათრეინე y-ის ნაცვლად log(y)-ზე რადგან rmse-ს ლოგარითმზე ითვლიან შეფასებისას.
    ასევე კორელაციის და კატეგორიულების treshold-ები (ჰიპერპარამეტრები) ხელით გადავარჩიე კონსტანტების შეცლვით.
    ასევე ხელით გადავარჩიე მედიანის თუ საშუალოს აღება ჯობდა.
    #### **შედეგი და დასკვნა**
    ამ მოდელმა შედარებით კარგი შედეგი გამოიღო. ჩემი აზრით ყველაზე დიდი შედეგი გამოიღო log(y)-ზე ტრენინგმა და one_hot_encoding-მა და მიახლოებით 0.14-ზე ჩამოიყვანა rmse. kaggle-ზე ეს 0.16 ზე ადიოდა საბოლოოდ რაც დიდი ალბათობით train-ზე overfitting-ის ბრალია.
    
3. ### **LinearRegression scaling + rfe + logs + col_X_col + one_hot**
    >[ექსპერიმენტის ლინკი](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/4)
    #### **მიმოხილვა** 
    ამ მოდელში წინა მოდელებთად შედარებით უფრო კომპლექსური მიდგომები გამოვიყენე და უფრო მეტი შანსი მივეცი მოდელს რომ კარგად შეესწავლა მონაცემები. scaling-ს საბოლოო ვერსიაში აღარ ვიყენებდი რადგან ცუდ შედეგს მაძლევდა. ეს ალბათ იმის ბრალია რომ ახალ დამატებულ 
    სვეტებსა და ძველებს შორის მაგნიტუდის დიდი ცვლილება აღარ არის ისე დიდი და ნაკლების მომცემია მოდელისთვის scaler-ის გამოყენების შემდეგ
    #### **Feature Engineering**
    რიცხვებში ისევ საშუალოებით ჩავანაცვლე. დამატებით ვცადე დამემატებინა სვეტები 10 ყველაზე კარგ რიცხვით სვეტების წყვილ-წყვილად გადამრავლებული სვეტები. კიდევ დავამატე სვეტების ლოგარითმები. ამ დამატებული სვეტებით მინდოდა მოდელისთვის უფრო მეტი ინფორმაცია მიმეცა რომლებიდანაც საუკეთესოებს აირჩევდა.კატეგორიულებიდან კი ყველა one_hot_encoding-ით გავშალე რადგან feature selection-ს მივანდე სვეტების გადაყრა. საბოლოოდ  ყველაფერი  standard scaler-ით დავასკალირე.
    #### **Feature Selection**
    წინა მიდგომასავით გამოვიყენე კორელაციის ფილტრი და დამატებით rfe-ს გამოყენებით გადავაგდე სვეტები.
    #### **Training**
    Linear Regression წინა მოდელივით დავატრენინგე და ჰიპერპარამეტრებს ხელით ვცვლიდი.
    #### **შედეგი და დასკვნა**
    ამ მოდელმა არ გამოიღო ისეთი კარგი შედეგი ვალიდაციაზე როგოც ველოდი და წინასთან შედარებით ცოტა უარესი იყო rmse(~0.15) სამაგიეროდ ეს შედეგი იყო kaggle-ზეც წინას საბოლოო შედეგზე უკეთესია. გამოვიდა რომ რაც აქ ისწავლა არ იყო დაზეპირება და იყო ზოგადი კანონზომიერები და ამის გამო დაჯდა იგივე შედეგი kaggle-ზე.
4. ### **RandomForestRegressor rfe + logs + col_X_col + one_hot**
    > [ექსპერიმენტის ლინკი 1](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/5)
    
    > [ექსპერიმენტის ლინკი 2](https://dagshub.com/adola23/HousePricesML.mlflow/#/experiments/6)
    #### **მიმოხილვა** 
    ამ დავალებას ჩემი აზრით DecisionTree არ შეეფერებოდა რადგან სახლის ფასები უფრო წრფივ ამოცანას გავს რადგან მარტივად შეიძლება ორი სახლის შედარება უბრალოდ რიცხვობრივი მეტობებით მაგალითად ფართობი, ოთახების რაოდენობა და მსგავსი რიცხვები. ამიტომ დიდი დრო არ დამითმია DecisionTree-ებისთვის და წინა ექსპერიმენტის preprocessing-ით ვცადე მენახა მიახლოებით რას მოგვცემდა ამ ტიპის მიდგომა.
    #### **Feature Engineering**
    იგივე რაც წინა ექსპერიმენტი.
    #### **Feature Selection**
    იგივე რაც წინა ექსპერიმენტი.
    #### **Training**
    DecisionTree-ც და RandomForest-იც აქ დავატრენინგე და ტესტის და ტრენინგსი rmse დავლოგე მოდელების შესაფასებლად. პარამეტრებს ხელით ვცვლიდი. თავიდან overfitting ხდებოდა როდესაც ხის სიღრმე არ მქონდა შეზღუდული რადგან ტრენინგზე log rmse მიახლოებით 0 იყო ხოლო ვალიდაციაზე ~0.22. სიღრმის და სხვა პარამეტრების შეცვლამ overfitting-ს უშველა მაგრამ ვალიდაცეაზე 0.2-ს მაინვ ვერ ჩამოცდა.
    #### **შედეგი და დასკვნა**
    DecisionTree ამ ტიპის ამოცანებისთვის საუკეთესო არ არის და LinearRegression უკეთესია.

# საბოლოო შედეგი
საბოლოოდ ავირჩიე მე-3 მოდელი **0.15033 rmse**-თი რადგან საუკეთესო შედეგი დადო kaggle-ზე და ლოკალური rmse თითქმის ზუსტად დაემთხვა kaggle-ზე დაფიქსირებულს რაც იმას ადასტურებს რომ ამ მოდელმა ზოგადად ეს ამოცანა ისწავლა ამ დონეზე. სხვა მოდელებზე ძირითადად overfitting-ხდებოდა და ამის შემცირებით დიდად არ უმჯობესდებოდა შედეგი. underfitting მგონი არ შეგვხვდა ამ ექპერიმენტებში რადგან ჩემი ძირითადი მიდგომა იყო ბევრი ინფორმაციის მიცემა, რომ მოდელს თვითონ აერჩია საუკეთესო ინფორმაცია და ნელ-ნელა ვაშორებდი ინფორმაციას სანამ კარგ შედეგამდე არ მივიდოდა რაც ნიშნავს რომ თავიდან უფრო მეტი nose-იყო და ხდებოდა უმნიშვნელო ინფორმაციაზე ყურადღების გამახვილება.
